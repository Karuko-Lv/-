### Cache 系统的性能评价
#### Cache 系统
![[截屏2026-06-16 20.53.51.png]]
假设 CPU 要进行数据访问，为了简单起见假设数据访问为读操作 
1. CPU 首先在 cache 中查找待访问的数据（通过某种查找机制实现）
1️⃣找到了
2. 数据在 cache 中则 called 数据命中 
3. 将命中的数据读取到 CPU 即可
上述数据访问时间 called 命中访问时间,记为 $t_c$
包括 1️⃣cache 内查找时间 2️⃣cache 访问时间
![[截屏2026-06-16 20.58.26.png]]
2️⃣没找到
2. CPU 在 cache 中未查找到所要访问的数据 called 数据缺失
3. 将缺失的数据从主存调入到 cache 中
4. CPU 从 cache 中访问数据
上述数据访问时间 called 缺失补偿
包括 1️⃣cache 内的查找时间 2️⃣主存访问时间记为 $t_m$ 3️⃣cache 访问时间
由于 $t_m$ 较大所以缺失补偿粗略记为 $t_m$ 
🧚‍♀️这里白中英的书说的是，用主存读周期把该字从主存送到 cpu，与此同时把含有该字的块送到 Cache；在 cache 中没有命中，CPU 也不去主存，而是将数据从主存调入 cache，CPU 再在 cache 中访问；对，redis 未命中直接查数据库返回，同时再写入 redis；块存储；prefetch

🧚‍♀️标记+索引+块内地址；绝对+相对；分页分段？地址加偏移；数据块内偏移是对cache的再分，比如我们想在第i个cache行写数据，不能随便找地方对吧，需要这个块内偏移量告诉我们写在这个cache行的那一部分这么理解对吗 uu们；实际上是一个三级定位；各位，一点要注意，s、r 和 w 是地址（二进制数）的位数！并不是区的数量、块的数量（十进制数）！；这里数据块大小应该是 2^w 个字，因为 cache 和 cpu 以字为数据交换单位；其实块内偏移也就是对数据块的编码，找到的东西是个字节也就乘字节了；按字节寻址，数据块内偏移位数里每一个二进制数可代表一个字节；这是按字节编址的，这些算的全是个数，不是字节或比特，所以最后要乘以八

直接映射方式下待访问信息的主存地址的构成👆，通过待访问信息的主存地址可以：
1️⃣在主存中精确定位该信息的位置

🧚‍♀️数据块内偏移就是用来说明需要的数据在数据块中的位置，在哪个字节，比如一个数据块有 8 个字节，块内偏移 000 就是第 0 个字节，111 就是第 7 个字节，所以块内偏移数也能说明数据块内有几个字节
🧑‍🏫假设红色块是主存中某个待访问的信息，属于分区 1，其主存地址的主存分区号部分的值是 1；因为该信息位于分区 1 的数据块 1，因此主存地址的区内数据块号（或称之为 cache 行号或索引）值为 1；假设该信息在数据块 1 中的起始位置距离数据块 1 的起始地址偏移了 8 个字节，因此主存地址的数据块内偏移部分的值为 8==>通过该**信息的主存地址**可以精确描述出该信息在主存中的具体位置
Q：如何通过主存地址在 cache 中进行快速查找
A：🧑‍🏫通过信息的区内数据块号即 cache 行号的取值，可以索引到该信息所在主存数据块所对应的 cache 行；直接映射规则可知主存中会有多个数据块对应 cache 中的某个固定行，例如本例中主存分区 0 的数据块 1 对应 cache 行 1、主存分区 1 的数据块 1 也对应 cache 行 1... 主存分区 m-1 的数据块 1 也对应 cache 行 1，❓如何判断 cache 行 1 中当前保存的主存数据块的副本是否是待访问信息所在的那个主存数据块呢❓❓❓
Cache 行中包含有主存分区号的作用：通过比较 cache 行中主存分区号的取值与待访问信息的主存地址中的主存分区号部分的取值是否相等，从而判断出 cache 行中的数据块副本保存的是否是待访问信息所在的主存数据块，如果相等则数据命中；不等则数据缺失。若该 cache 行中的有效位为 1 则表示该 cache 行信息有效，通过待访问信息的主存地址中的数据块内偏移部分的取值

🧚‍♀️一个数据块有很多字节，块内偏移就是指定一个块内的具体的某个字节的地址；因为每个块内还包含多条数据信息，块内偏移可以简单认为在块内再给每条信息编的地址；就是块内地址，他不会记录到 cache 中只是通过硬件，在找到对应的在 cache 的哪一块后，对比数据是否一致；有效位是看你主存的东西有没有放到 cache 中去，cache 中某一行已经存放了主存中的某一行数据，那么有效位就是 1；实际应该先检查有效位，然后检查分区号，不会出现缓冲命中但是不有效，不有效就是没命中。
