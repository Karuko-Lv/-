🧑‍🏫由于 CPU 有较强的程序局部性所以可以知道有经常访问的数据，可以复制到 cache 中，大部分数据在 cache 中访问就可以提高速度

### 程序局部性
![[截屏2026-06-16 20.32.30.png]]
🧚‍♀️是不是类似于操作系统的快表；缓存思想；空间局部性就是在时间 1 访问地址 1，那么他接下来就有可能会访问地址 2，3.. 就跟那个数组一样，你访问了 0 号元素那么很有可能会访问 1，2，3 号元素，这个就叫空间局部性连续地址访问；可以把时间局限性理解为恋旧，也就是会一直访问空间局限性理解为就近用了一个东西后就会使用它附近的东西![[截屏2026-06-16 20.32.54.png]]
🧚‍♀️王道这里是说访问顺序和存档顺序一致才有空间局部性 ‼️

### Cache 系统的性能评价
![[截屏2026-06-16 20.53.51.png]]
假设 CPU 要进行数据访问，为了简单起见假设数据访问为读操作 
1. CPU 首先在 cache 中查找待访问的数据（通过某种查找机制实现）
1️⃣找到了
2. 数据在 cache 中则 called 数据命中 
3. 将命中的数据读取到 CPU 即可
上述数据访问时间 called 命中访问时间,记为 $t_c$
包括 1️⃣cache 内查找时间 2️⃣cache 访问时间
![[截屏2026-06-16 20.58.26.png]]
2️⃣没找到
2. CPU 在 cache 中未查找到所要访问的数据 called 数据缺失
3. 将缺失的数据从主存调入到 cache 中
4. CPU 从 cache 中访问数据
上述数据访问时间 called 缺失补偿
包括 1️⃣cache 内的查找时间 2️⃣主存访问时间记为 $t_m$ 3️⃣cache 访问时间
由于 $t_m$ 较大所以缺失补偿粗略记为 $t_m$ 
🧚‍♀️这里白中英的书说的是，用主存读周期把该字从主存送到 cpu，与此同时把含有该字的块送到 Cache；在 cache 中没有命中，CPU 也不去主存，而是将数据从主存调入 cache，CPU 再在 cache 中访问；对，redis 未命中直接查数据库返回，同时再写入 redis；块存储；prefetch
#### 数据分块
![[截屏2026-06-16 21.06.04.png]]
🧚‍♀️数据块过大，载入 cache 中的数据块个数就会越少，越容易发生冲突导致替换；块大缺失损失大，需要消耗更多的时间从主存重新调入；数据块过大，，替换的时候替换出去的多，之前经常被使用的也可能会替换出去，所以破坏了时间局部性；redis 过大并行差![[截屏2026-06-16 21.09.08.png]]
🧑‍🏫数据分块后需要给主存的数据块和 cache 中的数据块分配地址
![[截屏2026-06-16 21.10.47.png]]
Cache 内的块地址长度小于主存内的块地址长度
