### 树的逻辑结构
树🌲是一种递归定义的数据结构，要么是空树，要么至少有一个根结点，而根结点的下方可能会有 0 颗或者多颗互不相交的子树
![[截屏2026-05-10 16.33.49.png]]
### 树的存储 1: 双亲表示法（顺序存储）
二叉树的数组下标可以反映结点之间的逻辑关系
结点编号和完全二叉树相对应
![[截屏2026-05-10 17.17.18.png]]
但是无法推广到普通的树，每个分支结点的子树数量不确定
有完全二叉树但是没有完全数
基于**结点有且仅有一个双亲**，记录结点的父结点编号
==>**思路：** 用数组顺序存储各个结点，每个结点中保存数据元素、指向双亲结点（父结点）的“指针”
![[截屏2026-05-10 17.21.18.png]]
「PS」记录结点的总数 `int n;`
🧚‍♀️树状数组；静态链表吗；并查集雏形？；邻接表；这和树状数组的关系只有他俩都是树了；有点像静态链表；现在可以一对 n，静态链表是 1 对 1；不是指针吧应该是伪指针；那岂不是只能从子节点快速查找父节点，但是很难反过来；感觉应该叫单亲表示法
#### 双亲表示法存储森林
双亲表示法用-1 表示根结点
![[截屏2026-05-10 17.22.11.png]]

#### 优缺点
找一个结点的孩子结点只能将数组遍历一遍看哪些元素的 `parent` 双亲指针是指向这个结点的
![[截屏2026-05-10 17.24.22.png]]
‼️适用于“找父亲”多，“找孩子”少的应用场景==>并查集

### 树的存储 2: 孩子表示法（顺序➕链式）
还是用数组顺序存储各个结点的值，每个结点对应一个数组下标
由于一个结点可能有多个孩子
因此每个数据元素当中可以保留一个链表的头指针 `*firstchild`, 如果结点有孩子就在链表中保存所有孩子的编号 
==>**思路：** 用数组顺序存储各个结点，每个结点中保存数据元素、孩子链表头指针
![[截屏2026-05-10 17.30.28.png]]
```cpp
struct CTNode{
	int child;//孩子结点在数组中的位置
	struct CTNode *next;//下一个孩子
};
typedef struct{
	ElemType data;
	struct CTNode *firstChild;//第一个孩子
}CTBox;
typedef struct{
	CTBox nodes[MAX_TREE_SIZE];
	int n,r;//结点数和根的位置
}CTree;

```
🧚‍♀️算法题里边的哈希拉链法；邻接表；拉链法；❓既然是保存孩子节点的编号，为什么不用 `int*`，而用 `CTNode*` 呢==>要用带 next 指针的 CTNode（Child Tree Node）
![[截屏2026-05-10 17.35.35.png]]
「PS」要记录根结点的下标编号是多少!!!记录根结点的位置很重要‼️（一棵树的访问通常从根结点开始）
#### 孩子表示法存储森林
![[截屏2026-05-10 17.36.54.png]]
‼️需要在结构体中记录多个根的位置

#### 优缺点
![[截屏2026-05-10 17.40.17.png]]
找父亲只能依次遍历每一个链表
‼️适用于“找父亲”少，“找孩子”多的应用场景==>服务流程树（e.g拨打 10086 到选中某一个具体的服务）
🧚‍♀️那这个结构找双亲的时间复杂度是不是 O (mn) 啊？，n 是总节点数，m 是横向链表长度
#### 孩子🆚双亲表示
父节点只有一个不需要指针一个 int 变量就行
「总结」大多数的数据结构都不可能是完美的。既有优点也一定会有缺点，关键在于将数据结构放在适合的应用场景之中

### 树的存储 3: 孩子兄弟表示法（链式）
每一个结点用 struct 结构体保存，包含结点的数据和两个指针，其中一个指向当前结点的第一个孩子、另一个指向当前结点的右边的一个兄弟
和二叉树结点定义类似
![[截屏2026-05-10 17.46.58.png]]
🧚‍♀️sibling，同胞
![[截屏2026-05-10 17.48.24.png]]
用孩子兄弟表示存储，从存储的视角来看和二叉树的形态类似
🧚‍♀️这不是树转二叉树的方法吗? 左孩子右兄弟；脑子向右偏转 45 度，发现这两个树一样；像斜着的层次遍历 
#### 孩子兄弟表示法存储森林
![[截屏2026-05-10 17.51.27.png]]
🧚‍♀️森林转二叉树

🧚‍♀️好处是什么？方便查找双亲还是方便查找孩子？；优点: 找孩子树转森林；这个适合层次遍历啊；只有一棵树的情况根的话是只有孩子，没有兄弟；为什么没有父亲孩子表示法呢
