## 三、状态维护方式

### 3.1 Pinia Setup Store 模式

项目采用 **Setup Function** 模式（而非 Options 模式）：

```typescript
export const useNaISpectrometerStore = defineStore("naiSpectrometer", () => {
  // 1. 定义状态
  const status = ref<NaISpectrometerStatus | null>(null)
  const spectrum = ref<NaISpectrum | null>(null)
  
  // 2. 定义计算属性（派生状态）
  const workStatus = computed(() => {
    const state = spectrometerState.value;
    if (state === 1) return "刻度中";
    if (state === 2) return "刻度成功";
    return "未开始";
  })
  
  // 3. 定义方法
  const fetchStatus = async () => { /* ... */ }
  const connectSSE = () => { /* ... */ }
  
  // 4. 暴露给外部
  return { status, spectrum, workStatus, fetchStatus, connectSSE }
})
```

### 3.2 计算属性（Computed）- 派生状态

```typescript
// 碘化钠谱仪工作状态转换
const workStatus = computed(() => {
  const state = spectrometerState.value;
  switch(state) {
    case 1: return "刻度中"
    case 2: return "刻度成功"
    case 3: return "刻度失败"
    default: return "未开始"
  }
})

// 气溶胶开关状态转换
const aerosolSwitch = computed(() => 
  aerosolState.value === 1 ? "开启" : "关闭"
)
```

**作用**：
- 避免组件重复计算
- 状态转换逻辑集中管理
- 响应式缓存，仅依赖变化时重新计算

```typescript
const fetchStatus = async () => {
  loadingStatus.value = true
  try {
    // 1. 调用API层

  }
```

```typescript
const clearDose = async () => {
```

### 4.2 Store ↔ SSE 层交互

#### SSE 连接管理（核心职责）
```typescript
const connectSSE = () => {
  // 1. 防重复连接
  if (isConnected.value) {
    console.log("SSE已连接")
    return
  }

  try {
    // 2. 建立EventSource连接
    const sseUrl = "/api/sse/qix"
    eventSource = new EventSource(sseUrl)
    
    // 3. 监听数据事件
    eventSource.addEventListener('device.data', (event) => {
      try {
        // 4. 数据清洗
        let cleanData = event.data.replace(/^\ufeff/, '')
        
        // 5. 数据解析
        const data = JSON.parse(cleanData)
        
        // 6. 数据转换与更新
        updateWeatherData(data)
        
        // 7. 趋势数据维护（滑动窗口）
        const timestamp = new Date().toLocaleTimeString()
        if (data.temperature) {
          updateTemperatureTrend(timestamp, parseFloat(data.temperature))
        }
      } catch (error) {
        // 8. 容错处理
        console.error("解析数据失败:", error)
        // 降级方案：尝试eval解析
        try {
          const evalData = eval('(' + event.data + ')')
          updateWeatherData(evalData)
        } catch (evalError) {
          console.error("降级解析也失败:", evalError)
        }
      }
    })
    
    // 9. 监听错误
    eventSource.addEventListener('error', (error) => {
      console.error("连接错误:", error)
    })
    
    isConnected.value = true
  } catch (error) {
    console.error("建立连接失败:", error)
  }
}
```

#### 双 SSE 连接 - 碘化钠谱仪（高级模式）
```typescript
// 连接1：主数据（剂量率、状态等）
dhnEventSource = new EventSource("/api/sse/dianhn")

// 连接2：能谱数据（1024道，数据量大）
dhnSpectrumEventSource = new EventSource("/api/sse/dianhn/spectrum")
```

**设计优势**：
- 数据分流，避免单一连接阻塞
- 能谱数据量大，独立连接更稳定
- 可以独立控制不同数据流

### 5.2 数据清洗策略（多层容错）

#### 层级 1：基础清洗
```typescript
// 去除BOM字符
let cleanData = event.data.replace(/^\ufeff/, '')

// 修复常见JSON格式错误
cleanData = cleanData.replace(/"deviceId":\s*"([^"]*),/g, '"deviceId": "$1",')
cleanData = cleanData.replace(/\x00/g, '0')
```

#### 层级 2：标准解析
```typescript
try {
  const data = JSON.parse(cleanData)
  // 正常处理
} catch (jsonError) {
  // 降级处理
}
```

#### 层级 3：降级解析（eval 作为备选）
```typescript
try {
  const evalData = eval('(' + event.data + ')')
  console.log("eval解析成功:", evalData)
  // 使用eval解析的数据
} catch (evalError) {
  console.error("所有解析方式都失败")
}
```

### 5.3 趋势数据维护（滑动窗口）

```typescript
const updateTemperatureTrend = (timestamp: string, value: number) => {
  // 1. 检查窗口大小
  if (temperatureTrend.timestamps.length >= 10) {
    // 2. 移出最早的数据
    temperatureTrend.timestamps.shift()
    temperatureTrend.values.shift()
  }
  
  // 3. 添加新数据
  temperatureTrend.timestamps.push(timestamp)
  temperatureTrend.values.push(value)
}
```

**特点**：
- 固定窗口大小（10 个数据点）
- 先进先出（FIFO）
- 内存占用稳定

### 5.4 复杂数据处理 - 1024 道能谱

```typescript
const updateSpectrumData = (data: any) => {
  // 1. 初始化数组
  const spectrumChannels: number[] = new Array(1025).fill(0)
  
  // 2. 遍历所有键，支持多种命名格式
  Object.keys(data).forEach(key => {
    // 格式1: spectrum_channel_0, spectrum_channel_1, ...
    if (key.startsWith('spectrum_channel_')) {
      const channelMatch = key.match(/spectrum_channel_(\d+)/)
      if (channelMatch) {
        const channel = parseInt(channelMatch[1], 10)
        if (channel >= 0 && channel <= 1024) {
          spectrumChannels[channel] = Number(data[key])
        }
      }
    }
    // 格式2: spectrumChannel0, spectrumChannel1, ...
    else if (key.startsWith('spectrumChannel')) {
      const channelMatch = key.match(/spectrumChannel(\d+)/)
      // ...处理
    }
    // 格式3: channel0, channel1, ...
    else if (key.startsWith('channel')) {
      // ...处理
    }
  })
  
  // 3. 更新状态（双份更新，确保兼容）
  spectrumData.value = spectrumChannels
  spectrum.value = {
    size: 1024,
    counts: spectrumChannels
  }
}
```

```

   ↓           ↓            ↓
```

**优势**：

## 七、职责范围边界

### 7.1 Store 应该做的 ✅

| 职责 | 说明 |
|------|------|
| **数据存储** | 所有业务状态、设备数据 |
| **响应式管理** | ref、reactive、computed |
| **SSE 连接** | 建立、监听、断开、错误处理 |
| **API 调用** | 封装 HTTP 请求、loading 状态 |
| **数据清洗** | JSON 解析、格式修复、容错 |
| **业务逻辑** | 状态转换、数据验证、计算 |
| **错误处理** | 统一错误消息、用户提示 |

### 7.2 Store 不应该做的 ❌

| 职责 | 说明 | 应该放在哪里 |
|------|------|-------------|
| **DOM 操作** | 直接操作 DOM | 组件 |
| **样式计算** | 动态样式 | 组件 computed |
| **路由跳转** | 页面导航 | 组件/router |
| **模板逻辑** | v-if、v-for | 组件模板 |
| **用户交互** | 点击、输入处理 | 组件方法 |

## 八、核心作用总结

### 8.1 对开发者的价值

1. **代码组织清晰**：业务逻辑集中，易于维护
2. **状态共享简单**：跨组件数据共享无需 props/events
3. **测试友好**：独立的业务逻辑，易于单元测试
4. **开发效率高**：响应式系统自动处理 UI 更新

### 8.2 对应用的价值

1. **数据一致性**：单一数据源，避免不同步
2. **实时性保证**：SSE 连接管理，数据秒级更新
3. **容错性强**：多层数据清洗，降级方案
4. **性能优化**：计算属性缓存，避免重复计算

### 8.3 架构模式价值

1. **关注点分离**：UI 与业务逻辑解耦
2. **可扩展性**：新增设备只需新增 Store
3. **可维护性**：代码结构清晰，职责明确
4. **可复用性**：Store 可在多个组件间复用

#### 预设场景题
1️⃣ 既然你用了 Axios 封装和 SSE，那如果你的后端同学突然想把所有 SSE 改成 WebSocket，你的前端代码改动大吗？

- 答：“改动会非常小。因为我采用了‘接口与实现分离’的策略。在 `deviceStore` 中，业务逻辑只关心 `updateData()` 这个方法，而不关心数据是怎么来的。我只需要写一个新的 `WebSocket` 服务类，替换掉原来的 `SSE` 服务类，然后把解析后的数据依然喂给同一个 Store 即可

理解：**接口与实现分离**体现为上层业务逻辑只关心“我要做什么”，而不关心“底层是怎么做到的”。即Vue 组件（界面）和 Pinia Store（大脑）不应该知道数据是通过 Axios 拿到的，还是通过 SSE 推送过来的。

2️⃣ 实时数据渲染方案：从“直接驱动”到“数据缓冲池”
